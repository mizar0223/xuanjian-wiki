## 3. 架构与铁律

### 模板与字段
- **统一模板**：单一 `{{造物}}` 模板覆盖全造物类
- **道统必填**：每个实体需填道统（✅ 明确 / ⚡ 推断 / ❓ 未知）
- **品阶三同步**（P0 铁律）：info 表品阶 / 首句"是…的[[X]]" / Category 三处必须一致

### 内容生产
- **宽窗拼接**：每次提及前后各 500 字 → context.txt
- **深度推理**：单页深度（名字由来 + 功能与威能 + 外观 + 历史 + 相关实体）≈ 3 min/页
- **品阶并列推断**：原文出现"X、Y、Z"且 Y/Z 已知品阶时，X 同品阶

### 内链规范（v1）
`[[造物-XX]] [[人物-XX]] [[势力-XX]] [[地点-XX]] [[道统-XX]] [[神通-XX]] [[事件-XX]]`

### 数据可信度标记
- ✅ 明确（原文证据）
- ⚡ 推断（间接证据）
- ❓ 未知（不猜测）

## 9. Lessons Learned（重要教训）

1. **batch_generator 会覆盖 agent 输出** → MUST 先检查文件大小
2. **文件名 ≠ 页面标题**：`L4-XX.wiki` 不能直接当 stem 用，必须脱前缀
3. **Wiki .env 必须 source**：`set -a && source .env && set +a`
4. **upload_staging 是死分支**：不要再回写，pages 是唯一源
5. **3 agent 是硬上限**：超过会僵尸+context 膨胀+决策退化
6. **目录品类 ≠ 原文品阶**：B11-B15 发现 4 处早期错误（戊咲珠/缡刃/牝水/飞玄乱石）
7. **品阶并列推断要谨慎**：大夏郢铜甲证据不足曾误推，已回退
8. **MediaWiki URL 格式**：线上地址永远用 `https://leoshixie.devcloud.woa.com/wiki/页面名`，**HTTPS + 禁止加 `index.php`**。中文标题会自动 URL 编码。该环境启用了 URL rewrite，加 index.php 或 HTTP 都会出错。2026-05-11 已犯两次。
9. **00-体系目录的专题页不应加造物前缀**：上传时必须用 `--no-prefix`，否则会创建 `造物-灵宝` 之类的错误页面。已犯错并删除修复。
10. **删除页面用 API**：`action=delete` + CSRF token，需用 `clientlogin` 认证，API endpoint 是 `http://leoshixie.devcloud.woa.com/api.php`（非 /wiki/api.php）。
11. **宽窗 context 优先原则（v3.1）**：wiki 页面的【原文引用】是我挑选的二手资料（仅2-3条），宽窗 context.txt 是从39个原著txt提取的一手原始资料（500+500字完整窗口）。必须先通读 context 再对照 wiki 引用区，顺序颠倒会导致关键证据（如第944章《瑞成》、青衍「瑞炁之丹」）被遗漏。2026-05-12 望晋玄衍丹错误即因此。
12. **持有者道统陷阱（再次出现）**：都道铃原文明确说「是『清炁』」，但名字含「都道」+ 持有者是都仙道少主 → 被误标为都仙。持有者道统 ≠ 法器道统，原文有直接定性时必须以原文为准。
13. **章名是证据**：第944章《瑞成》的「瑞」字是瑞炁道统的关键线索，作者通过章名给出最精炼概括，不可忽略。
