# 玄鉴仙族 Wiki 竞品深度分析与迭代 roadmap

> 分析日期: 2026-05-07
> 分析范围: 网络小说/历史/人物传记类 Wiki 平台
> 标杆案例: 灰机 Wiki、萌娘百科、Fandom、百度百科星图、掌门路 Fandom

---

## 一、执行摘要

玄鉴 Wiki 目前处于**"内容填充期"向"功能深化期"**过渡的阶段。354 个页面 + 73 张人设图 + SMW 语义引擎已就位，但在**数据可视化、模板丰富度、移动端体验、社区互动**四个维度与市面标杆存在明显差距。

**核心结论**: 玄鉴 Wiki 的差异化优势在于**家族修仙题材天然的谱系/势力/时间线数据密度**，这是单人主角流小说 Wiki 不具备的。应围绕"**关系可视化 + 时间线叙事 + 势力博弈图谱**"打造差异化体验，而非追求大而全。

### 4.2 题材适配度分析

玄鉴作为**"家族修仙群像小说"**，与其他题材的 Wiki 需求差异:

| 题材类型 | 核心数据维度 | Wiki 侧重点 | 玄鉴适配度 |
|---------|------------|-----------|----------|
| 单人升级流（凡人修仙传） | 主角修为、功法、地图 | 个人成长时间线 | ⭐⭐⭐ |
| 群像权谋（冰与火之歌） | 家族、势力、婚姻、战争 | 关系网络 + 地理政治 | ⭐⭐⭐⭐⭐ |
| 克苏鲁/诡秘（诡秘之主） | 序列、魔药、组织、历史 | 体系化知识树 | ⭐⭐⭐⭐ |
| 家族修仙（玄鉴仙族） | **家族谱系、势力博弈、功法传承、地缘格局** | **谱系图 + 势力图谱 + 时间线** | ⭐⭐⭐⭐⭐ |

**结论**: 玄鉴 Wiki 应该向**"冰与火之歌灰机 Wiki"**的对标方向进化，而非向"凡人修仙传百科"方向。

## 六、具体改进建议

### 6.1 模板体系重构

**当前问题**: 只有 `{{角色}}` 一个模板，参数简单，无导航联动。

**建议架构**:
```
{{角色}}        → 人物基础信息 + 修为 + 灵根 + 势力归属
{{势力}}        → 门派/家族信息 + 从属关系 + 领地 + 现任掌门
{{功法}}        → 功法名称 + 属性 + 等级 + 修炼者列表
{{地点}}        → 地理信息 + 所属势力 + 相关事件
{{事件}}        → 时间 + 地点 + 参与角色 + 结果 + 影响
{{家族树}}      → 嵌入式的家谱关系图
{{势力导航}}     → 页面底部自动关联同势力角色
{{卷次导航}}     → 章节间的面包屑导航
```

### 6.2 SMW Property 设计

建议 Property 体系:

```
角色类:
  - Has name (Text)
  - Has alias (Text, 多值)
  - Has gender (Text)
  - Has cultivation (Text) — 修为境界
  - Has spirit root (Text) — 灵根属性
  - Has technique (Page, 多值) — 修炼功法
  - Has faction (Page) — 所属势力
  - Has status (Text) — 存活/死亡/失踪
  - Has family (Page, 多值) — 家族成员
  - Has master (Page) — 师父
  - Has apprentice (Page, 多值) — 徒弟
  - Has spouse (Page) — 道侣
  - First appearance (Text) — 首次出场章节
  - Death chapter (Text) — 死亡章节

势力类:
  - Has leader (Page)
  - Has location (Page)
  - Has type (Text) — 宗门/家族/散修
  - Has strength (Text) — 势力等级

地点类:
  - Has coordinates (Text)
  - Has ruler (Page)
  - Has type (Text) — 山脉/城池/秘境
```

### 6.3 家族谱系可视化方案

**方案 A: MediaWiki 原生**（推荐短期）
- 使用 `{{Familytree}}` 或 `{{Tree chart}}` 模板
- 优点: 无需额外依赖，编辑友好
- 缺点: 大型家谱性能差，样式受限
- 适用: 李家单一家谱（< 100 人）

**方案 B: D3.js 嵌入**（推荐长期）
- 在 wiki 页面嵌入 D3.js 力导向图
- 优点: 交互性强，可缩放/拖拽/筛选
- 缺点: 需要前端开发，SMW 数据导出为 JSON
- 适用: 全角色关系网、势力图谱

**方案 C: 外部工具 + iframe**
- 用 ECharts / Cytoscape.js 构建独立可视化页面
- 通过 iframe 嵌入 wiki
- 优点: 完全自由的交互设计
- 缺点: 维护成本高

### 6.4 移动端优化

Vector 2022 在移动端表现一般。建议:
1. **短期**: 调整 Vector 2022 的移动端 CSS，增大触摸区域
2. **中期**: 评估 MinervaNeue 皮肤（Wikipedia 移动端默认皮肤）
3. **长期**: 开发微信小程序/ H5 轻量版，聚焦"角色查询"和"家谱浏览"

### 6.5 内容深化策略

玄鉴 Wiki 目前最大的内容短板是**"角色与事件的关联缺失"**:

| 当前 | 目标 |
|------|------|
| "李通崖，筑基，李家第二代..." | "李通崖，筑基，李家第二代，于第 127 章突破筑基，参与第 234 章'望月之战'，于第 456 章坐化..." |
| 孤立的角色页 | 角色页 ↔ 事件页 ↔ 章节页 的三向链接 |

建议建立 **"事件中心"** 索引:
- 每个事件一个词条（如"望月之战"、"李家灭门危机"）
- 事件页列出: 时间、地点、参与角色、结果、影响
- 角色页自动显示"相关事件"列表（通过 SMW 查询）

## 七、与市面 Wiki 的核心差距总结

### 差距 1: 数据可视化（最大差距）
- **现状**: 纯文本 + 静态图片
- **标杆**: 灰机冰火维基有完整的家族树、地图、关系网
- **影响**: 玄鉴的核心魅力在于"家族谱系和势力博弈"，没有可视化等于自废武功
- **优先级**: P0

### 差距 2: SMW 语义化（技术已就绪，内容未跟上）
- **现状**: 安装了 SMW，但无 Property 定义，无查询页面
- **标杆**: 灰机冰火维基的语义化改造备忘录显示完整的 Property → Query → Format 链路
- **影响**: 无法做动态分类、自动关联、统计仪表盘
- **优先级**: P1

### 差距 3: 模板/导航体系
- **现状**: 单一角色模板，无导航框，页面间孤立
- **标杆**: 萌娘百科的模板规范体系，灰机的导航模板家族
- **影响**: 用户体验碎片化，难以探索关联内容
- **优先级**: P1

### 差距 4: 移动端与分发
- **现状**: 仅桌面浏览器访问，无小程序/App
- **标杆**: 灰机狗头人 App、百度贴吧小程序
- **影响**: 网文读者 70%+ 在移动端阅读，Wiki 也需要移动端友好
- **优先级**: P2
