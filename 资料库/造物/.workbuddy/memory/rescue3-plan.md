## 五、Agent 阻塞恢复协议

> **追加日期**：2026-05-10 | **来源**：Rescue 3 Batch 1→2 过渡阻塞实战教训

### 背景

Batch 1（36页）完成后，Batch 1→2 过渡时遭遇 Agent 系统阻塞：

- 旧 agent 完成工作后进程退出，但 team member 记录未自动清理
- `shutdown_request` 发往已死的 agent 无法得到确认
- 3-agent 硬上限被僵尸 agent 占满，新 agent 无法创建
- `TeamDelete` 参数校验失败（`root: must NOT have additional properties`）
- 主理人连续尝试 40+ turns 不同策略，上下文膨胀导致决策退化
- **最终损失**：仅完成 47/119 页（39.5%），剩余 72 页停滞

### 规则 1：TeamDelete 前必须先 ToolSearch

`TeamDelete` 是 deferred tool，**绝不直接调用**：

```
正确流程：
1. ToolSearch(queries=["TeamDelete"]) → 获取参数 schema
2. 按 schema 构造 params（只传声明参数，杜绝 undeclared 字段）
3. DeferExecuteTool(toolName="TeamDelete", params={按schema})
```

> ⚠️ 错误信息 `root: must NOT have additional properties` 意味着传了未声明的参数。该错误对 LLM 不可操作——唯一的修复方式是先 ToolSearch 查 schema。

### 规则 2：Shutdown 不等待确认，直接 TeamDelete

```
旧协议（脆弱）：
  shutdown_request → 等待 shutdown_response → TeamDelete

新协议（鲁棒）：
  shutdown_request（fire-and-forget）→ 直接 TeamDelete

理由：agent 完成工作后随时可能退出进程。
     与其死等一个不会来的 response，
     不如直接清理 team 资源。agent 进程在 TeamDelete 时被一并回收。
```

### 规则 3：Agent 创建失败处理——三次重试 + 上报

```
同一批次的 agent 创建（Agent 工具调用）：

  第 1 次失败 → 等待 10s → 重试
  第 2 次失败 → 检查 team 状态（残留 agent？容量满？）→ 修复后重试
  第 3 次失败 → 🛑 停止。向用户报告：

    📊 当前进度：已完成 X/119 页
    🚧 卡在：Batch N / Agent M
    ❌ 失败原因：{具体错误信息}
    🔧 恢复选项：
      a) 继续重试（清理旧 team 后重新创建 agent）
      b) 主理人降级直接执行当前批次（≤12 页）
      c) 暂停等待用户手动干预
```

### 规则 4：直接执行是正式后备，非 SOP 违规

以下条件**同时满足**时，主理人可直接执行页面升级：

1. agent 创建连续失败 ≥3 次
2. team 状态确认无法在 5 turns 内修复
3. 已向用户报告并获得默许（用户 2 分钟内未回复 → 视为默许）

直接执行约束：
- 每批 ≤12 页，严格按 rescue3-plan 原定批次顺序
- 每完成一页即 Edit 一页（不攒批）
- 完成后在 memory 记录，标注 `[主理人直接执行]`

### 规则 5：批次过渡检查清单

每批次完成后，逐项检查再启动下一批次：

```
□ 确认当前批次所有页面的文件修改时间 > 本次启动时间
  （find pages -name "*.wiki" -newer <启动标记文件> | wc -l）
□ 抽查 2-3 个页面：名字由来已填充、功能与威能已填充、铁律字段未改动
□ TeamDelete 旧 team（先 ToolSearch 查 schema！）
□ 确认旧 team 目录已删除：ls .workbuddy/teams/ 验证
□ TeamCreate 新 team（命名 rescue3-l5-b{N}）
□ 逐个创建 agent，每个间隔 ≥5s
□ 确认全部 agent 已回传任务确认（收到消息或文件已修改）
```

|------|------|------|
| Agent 创建报 "capacity" | 旧 agent 占槽未释放 | TeamDelete 旧 team（规则1） |
| TeamDelete 报 "additional properties" | 参数不符合 schema | ToolSearch → 重构造 params（规则1） |
| SendMessage 返回 "发送失败" | agent 进程已死 | 跳过，直接 TeamDelete（规则2） |
| Agent 创建成功但 300s 无产出 | agent 静默失败 | SendMessage 询问一次 → 无响应则重启 |
| 任意失败连续 3 次 | 系统级阻塞 | 🛑 上报用户（规则3） |

## 附录 B：Context.txt 格式说明

每个 context.txt 的头部格式：
```
# 《玄鉴仙族》【实体名】全量上下文
# 总提及: N次 | 合并窗口: M个 | 品类: XXX
# 窗口参数: 前500字+后500字
============================================================

↕↕↕ 窗口 1/M | 第XXX章《章名》 | 合并N个连续引用 ↕↕↕

[原文文本，实体名用【】标注]
```

- L5 页面的 context 通常有 1-2 个窗口
- 每个窗口约 2-6 KB
- 实体名在原文中用 `【】` 标注
- 窗口可能合并了多个相邻章节的引用
